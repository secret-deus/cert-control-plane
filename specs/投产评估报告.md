## BLOCKER 阻塞项（必须修复才能投产）

### B1. DEV_MODE 无生产护栏

**位置**：`server/app/api/agent.py:346-354`, `server/app/api/agent.py:388-396`

**问题**：当 `DEV_MODE=true` 时，heartbeat 和 fetch-certs 跳过所有 Agent 认证，任何未认证请求都能拉到任意 Agent 的私钥。虽然 `config.py` 默认值为 `False`，但没有任何机制阻止在生产 `.env` 中误开。

**风险**：一旦误配置，所有证书私钥对外暴露。

**修复建议**：

- 方案 A：当 `DATABASE_URL` 指向 PostgreSQL 时，拒绝启用 DEV_MODE，启动时抛异常。
- 方案 B：DEV_MODE 启动时打 CRITICAL 级别日志，且每次绕过认证时打 WARNING 日志。
- 方案 C：从代码中彻底移除 DEV_MODE，仅依赖测试 fixture 替代。

### B2. docker-compose.yml 是开发配置

**位置**：`docker-compose.yml`

**问题**：当前文件存在多个生产不可用的配置：

| 问题点 | 行为 | 生产风险 |
|--------|------|----------|
| `--reload` 标志 | 文件变更自动重启 | 性能损耗，意外重启 |
| 源码 bind-mount（`:ro`） | 覆盖镜像内构建产物 | 容器内运行的不是镜像代码 |
| 无 `restart: unless-stopped` | 进程崩溃后不自动恢复 | 服务中断 |
| app 服务无 health check | 编排器不知道 app 是否就绪 | 流量打到未就绪容器 |
| 无资源限制 | 无 mem_limit / cpus 限制 | 单容器耗尽宿主资源 |
| 默认 PostgreSQL 密码 `postgres` | 弱密码 | 数据库暴露风险 |

**修复建议**：

- 拆分为 `docker-compose.yml`（生产）和 `docker-compose.override.yml`（开发）。
- 生产版去掉 `--reload` 和 bind-mount，加上 `restart: unless-stopped`、health check（`/healthz`）、资源限制。
- PostgreSQL 密码从 `.env` 注入，不使用默认值。

### B3. 无限流（Rate Limiting）

**位置**：全局缺失

**问题**：所有端点均无限流保护。

| 端点 | 认证状态 | 暴露风险 |
|------|----------|----------|
| `POST /api/agent/register` | 无认证 | 洪泛创建百万条 pending 记录 |
| `GET /api/agent/register/status` | 无认证 | 枚举 Agent ID 和 fingerprint |
| `POST /api/control/*` | API Key 认证 | Admin API Key 暴力破解 |

**修复建议**：

- 接入 `slowapi`（FastAPI 限流中间件）。
- `/api/agent/register`：IP 级限流，如 10 次/分钟。
- Admin API：连续失败后 IP 级锁定。

### B4. 无指标/监控

**位置**：全局缺失

**问题**：没有 Prometheus、OpenTelemetry 或任何 metrics 集成。对于证书管理系统，缺少以下关键指标会导致运维盲区：

- 请求延迟 / 错误率
- 证书到期倒计时 gauge
- Agent 在线率
- Rollout 进度
- 数据库连接池使用率

**修复建议**：

- 接入 `prometheus-fastapi-instrumentator`，暴露 `/metrics` 端点。
- 新增业务指标：`certcp_certificate_expiry_seconds`（gauge）、`certcp_agents_online_total`（gauge）、`certcp_rollout_items_pending`（gauge）。

### B5. Go Agent PKCS8 解析 bug + 零测试

**位置**：`client/agent-go/internal/client/client.go:315`

**问题**：

1. `keyAny.(*interface{})` 类型断言几乎必然 panic。PKCS8 解析后应使用 `*rsa.PrivateKey` 等具体类型。
2. 整个 Go Agent 代码库没有任何 `_test.go` 文件，此 bug 无法被 CI 拦截。
3. 回滚逻辑不完整：chain 文件写入失败时不回滚已写入的 cert/key。
4. `copyFile` 始终设置 0644 权限，私钥备份文件权限过大。

**修复建议**：

- 如果 Go Agent 是发布目标：修复 PKCS8 bug、补 `_test.go`、修复回滚逻辑。
- 如果 Go Agent 仅为实验性实现：在 README 和 `docs/agent-comparison.md` 中明确标注"不用于生产"。

## PASS 项（已满足生产要求）

### 安全

- Fernet 加密私钥存储，启动时强制校验 `CA_KEY_ENCRYPTION_KEY` 非空
- `secrets.compare_digest` 时间安全比较 Admin API Key
- 全局 `Depends(verify_admin_key)` 保护 Control API 和 Dashboard API
- SQLAlchemy 参数化查询，零 SQL 注入风险
- 前端无 `dangerouslySetInnerHTML` / `innerHTML` / `eval`，零 XSS 风险
- API Key 使用 sessionStorage（关闭标签页即清除），不使用 localStorage
- 前端 API 请求通过相对路径 `/api/control`，无硬编码地址

### 架构与代码

- 模块划分清晰：`core/`、`api/`、`orchestrator/`、`registry/` 职责明确
- Pydantic v2 Schema 对用户输入字段有 `min_length` / `max_length` 约束
- 错误处理规范：无裸 `except:`，所有 `except Exception` 均有合理用途
- Rollout 编排器异常隔离：per-rollout catch + 独立 session 标记失败
- `PaginatedResponse` 泛型分页模式统一复用

### 测试与 CI

- 后端 150 个测试全部通过，含真实 SQLite 集成测试与 Kubernetes Secret 凭据更新 validate 覆盖
- 前端 `npm run lint`、`npm run build`、`npm run test:e2e` 全部通过
- 客户端 15 个测试全部通过，含 cert 部署 + rollback 路径测试
- CI 管道：5 job CI + 多平台 release + 6 种安全扫描（pip-audit、npm audit、Trivy、CodeQL、Gitleaks、dependency review）
- 安全扫描每周定时执行（周一 06:00 UTC）
- Playwright E2E 测试已配置

### 部署

- Dockerfile 多阶段构建（Node 22 → Python 3.12-slim）+ 非 root 用户
- Alembic 迁移链完整（001 → 005），有 downgrade 路径，migration 003 做了全量幂等对账
- minikube 部署脚本可用，已通过真实 K8s Secret E2E 验证

### 文档

- README 完整：架构概览、API 表格、时序图、部署指南、环境变量、安全设计
- 13 篇运维文档：部署、安全审计、预生产检查清单、告警、Agent 对比、发布模板
- OpenAPI / Swagger / ReDoc 自动生成，端点含双语 docstring
- 结构化 JSON 日志（`python-json-logger`），可配置日志级别
- 不可变审计日志（action / entity / actor / IP / timestamp）

## 三语言 Agent 功能对齐矩阵

| 功能 | Python | Go | Rust | 生产要求 |
|------|--------|-----|------|----------|
| TOFU 注册 | Yes | Yes | Yes | 必需 |
| 审批轮询 | Yes | Yes | Yes | 必需 |
| Heartbeat | Yes | Yes | Yes | 必需 |
| Fetch-certs | Yes | Yes | Yes | 必需 |
| 部署 + 回滚 | **Yes** | 部分 | **No** | 必需 |
| Report-certs | No | **Yes** | No | 建议 |
| Nginx 自动检测 | No | **Yes** | No | 可选 |
| InsecureSkipVerify | No | **Yes** | No | 测试用 |
| RSA 密钥长度 | 3072 | 3072 | **2048** | 应统一 3072 |
| 单元测试 | 2 个（deployer） | **0** | 3 个（crypto） | 必需 |

**结论**：仅 Python Agent 满足生产要求。Go Agent 有阻塞 bug，Rust Agent 缺少回滚。如需多语言支持，建议分阶段发布。
